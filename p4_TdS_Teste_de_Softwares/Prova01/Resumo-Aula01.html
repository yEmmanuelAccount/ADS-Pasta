O Processo de Software e suas Fases

O processo de software é definido como um conjunto de atividades que leva à produção de um produto (software).

As quatro fases principais do Processo de Software são:

  • Especificação de software.
  • Projeto e Implementação.
  • Validação de software.
  • Evolução de software.

Especificação de Software (Engenharia de Requisitos)

A especificação de software ou engenharia de requisitos é o processo necessário para compreender e definir quais serviços são necessários e identificar as restrições de cada operação.

Este estágio é considerado crítico, pois erros cometidos nessa fase conduzem a problemas posteriores no desenvolvimento.

O resultado da especificação é, em geral, um documento de requisitos que representa a especificação do sistema.

Fases da Especificação de Requisitos

  • Estudo de viabilidade: Ajuda a determinar se é possível produzir o que o cliente deseja com as opções de hardware, software, tempo e orçamento disponíveis.
  • Elicitação e análise de requisitos: Envolve a observação de sistemas existentes, análise de tarefas e discussão com clientes, ajudando o analista a compreender o sistema.
  • Especificação de requisitos: Consiste na elaboração de um documento com os requisitos coletados na fase anterior.
  • Validação de requisitos: Verifica se o documento está consistente. Os clientes leem os requisitos para verificar se atendem às suas necessidades.

Usuários do Documento de Requisitos

Como cada pessoa usa o Documento de Requisitos:

  • Clientes: Verificar se os requisitos atendem às suas necessidades e podem especificar mudanças.
  • Gerentes: Planejar um pedido de proposta para o sistema e planejar o processo de desenvolvimento.
  • Desenvolvedores: Compreender como o sistema será desenvolvido.
  • Testadores: Desenvolver testes de validação.

Tipos de Requisitos

  • Requisitos de Usuário:
    • São declarações, em linguagem natural com diagramas;
    • Sobre os serviços esperados e as restrições sob as quais o sistema deve operar;
    • São facilmente interpretados pelo usuário;
  • Requisitos de Sistema:
    • Detalham as funções, serviços e as restrições operacionais do sistema;
    • É uma descrição detalhada da funcionalidade e pode fazer parte do contrato entre o comprador e os desenvolvedores;
  • Requisitos Funcionais:
    • Descrevem o que o sistema deve fazer;
    • Detalham as funções do sistema, suas entradas e saídas, exceções, etc.;
    • Exemplo: Um usuário deve ser capaz de pesquisar listas de agendamentos;
  • Requisitos Não Funcionais:
    • Não estão diretamente relacionados com os serviços específicos oferecidos aos usuários, mas sim com as propriedades do sistema (confiabilidade, tempo de resposta, ocupação de área);
    • Eles especificam ou restringem as características do sistema como um todo;

Categorias e Exemplos de Requisitos Não Funcionais:

  • Desempenho: Especificam ou restringem o comportamento do software, como a rapidez com que o sistema deve executar uma função.
  • Organização: Derivados das políticas e procedimentos da organização, como requisitos do processo de desenvolvimento ou normas de processo a serem usadas.
  • Externos: Derivam de fatores externos ao sistema, como requisitos legais (para operar dentro da lei) e requisitos éticos (para ser aceitável para o público).

Atributos de Qualidade

  • USABILIDADE: Relacionada à facilidade de aprender (tempo e esforço mínimo exigido para o desempenho) e facilidade de uso (velocidade de execução de tarefas e redução de erros).
  • MANUTENIBILIDADE:
    • É a facilidade de fazer alteração;
    • Define a probabilidade de o software não causar uma falha em um sistema durante um determinado período de tempo sob certas condições;
    • É uma atividade de reparo de defeito, alteração/evolução de características existentes, ou adição de novas funcionalidades;
    • Relaciona-se com a arquitetura.
  • CONFIABILIDADE: Inclui conceitos como Disponibilidade (quão disponível o sistema estaria para uso), Taxa de ocorrência de falha (frequência de comportamento inesperado) e MTTF (Tempo médio até a ocorrência de falha).
  • DESEMPENHO: Refere-se à restrição sobre o comportamento do software, como a rapidez com que o sistema deve executar uma determinada função.
  • PORTABILIDADE: Facilidade na qual o software pode ser transferido de um sistema computacional ou ambiente para outro.
  • SEGURANÇA: Garante que acessos não autorizados ao sistema e dados associados não serão permitidos.

Validação e Verificação (V&V)

V&V busca responder duas perguntas principais:

  • “Estamos construindo o produto correto?” (Assegurar que o sistema atenda às expectativas do cliente).
  • “Estamos construindo o produto corretamente?” (Verificar se o software está de acordo com as especificações, atendendo aos requisitos funcionais e não funcionais).

Técnicas de V&V:

  • Inspeções ou revisões por pares: Representam uma análise estática e são baseadas no exame visual de documentos para detectar defeitos.
  • Testes de software:
    • Representam uma análise dinâmica, envolvendo a execução do programa com dados de testes;
    • É a principal técnica de V&V;
    • Tem como objetivo descobrir defeitos ou inadequações do programa por meio da análise das saídas, exercitando o programa simulando dados reais;

Terminologia Central de Teste

  • Ambiente de teste: Contém hardware, instrumentação, simuladores, ferramentas de software e outros elementos de suporte necessários à realização de um teste.
  • Automação da execução de teste: Utilização de software para controlar a execução dos testes, a comparação de resultados reais e esperados, o estabelecimento de pré-condições e outras funções de controle.
  • Caso de teste: Conjunto de valores de entrada, pré-condições de execução, resultados esperados e pós-condições desenvolvidos para um objetivo.
  • Cobertura: Grau (em porcentagem) que indica o quanto um item de cobertura foi exercitado por uma suíte de testes.
  • Código morto: Código inacessível.
  • Critério de aceite: Critérios de saída que um componente ou sistema deve satisfazer para ser aceito por um usuário, cliente ou outra entidade autorizada.
  • Dados de teste: Dados existentes (em um banco de dados, por exemplo) antes do início da execução de um teste e que afetam ou são afetados pelo sistema.
  • Domínio: Conjunto a partir do qual valores válidos de entrada e/ou saída podem ser selecionados.
  • Especificação de caso de teste: Documento que especifica um conjunto de casos de teste (objetivos, entradas, ações, resultados esperados) para um item de teste.
  • Integração: Processo de combinação de componentes ou sistemas em conjuntos maiores.
  • Não conformidade: Trata-se do não atendimento a um requisito especificado.
  • Plano de teste: Documento que descreve o escopo, a abordagem, os recursos e o cronograma das atividades de teste. Identifica itens de teste, tarefas, executor, ambiente, técnicas de projeto de teste e critérios de entrada/saída, além de riscos.
  • Qualidade de software: Totalidade das funcionalidades e características que se baseia na sua habilidade de satisfazer as necessidades declaradas ou implícitas.
  • Rastreabilidade: Habilidade de identificar itens relacionados em documentos e softwares, como requisitos e testes associados.
