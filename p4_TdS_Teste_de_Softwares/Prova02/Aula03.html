Conceitos Básicos

Definição

É o processo de executar um programa ou sistema com a intensão de encontrar erros.
É a simulação de um artefato executável com entradas e pre-condições conhecidas. A comparação entre a saída atual e a saída esperada determina se o artefato está em conformidade com os requisitos.
É o processo de executar um sistema ou componente dob condições específicas, observando e registrando os resultados, avaliando alguns aspectos do sistema ou componente.

Objetivo

O objetivos dos teste é encontrar defeitos e nunca afirmar que um sistema não possui erros.

  1. Um bom teste não é redundante. Cada teste deve ter uma finalidade.
  2. Um bom teste não deve ser muito simples e também nem muito completo.
  3. Encontrar o maior número de erros com o mínimo de esforço (custo/tempo).
  4. Testar todas as possibilidades?

Caso de Teste

Conjunto de dados de entrada, condições de execução do sistema e o resultado esperado/desenvolvido para atingir um objetivo específico.

O objetivo pode ser:

  • exercitar um caminho específico em um programa;
  • verificar a concorrência do programa com o requisito;

Nomenclaturas

  1. Identificador do caso de teste: Número único de identificação do caso de teste em questão.
  2. Objetivo do Caso de Teste: Mostra de forma breve qual é a intenção de teste do caso de teste.
  3. Entradas: Todas as entradas que serão consideradas no caso de teste.
  4. Saídas: Resultados esperados ao final do teste de acordo com as entradas indicadas na execução.
  5. Ambientes necessários / Condições de ambiente: O contexto de hardware, software e versão usados para a realização do teste. Importante para a replicação e cálculos de escalonamento ou para a identificação de problemas.
  6. Procedimentos especiais requisitados para a execução do caso de teste: Quais procedimentos devem ser adotados para o Analista de Testes executar o teste. Deve especificar com detalhes o que deve ser testado e a forma que o teste deve ser executado.
  7. Dependência com outros Casos de Teste: Caso esse Caso de Teste estar relacionado com outro, então o relacionamento entre eles deve ser explicitado.

Exemplo 01: Números Negativos

O sistema deve receber um valor inteiro, subtrair um desse valor e, após essa operação, dividir o resultado por 30000.

            int exemplo_simples(int num) {
                num = num - 1;
                num = num / 30000
                return num;
            }

Elementos do Teste

  1. Identificador: Caso de Teste 01: entrada negativa.
  2. Objetivo: Verificar se entradas negativas são possíveis.
  3. Entrada: (E) Digite "-30000"
  4. Saída: (S) 0,99
  5. Ambiente: Nenhum
  6. Procedimento: Nenhum
  7. Dependência: Nenhuma

Exemplo 02: Login de Usuário

  1. Caso de Teste 01: Login
  2. Verificar login inválido
  3. Entradas [E]
  4. Saídas [S]
    1. [E] No celular, clique no ícone do aplicativo;
    2. [S] O aplicativo deve abrir e exibir a tela de login;
    3. [E] Insira um email inválido, sem o sinal de "@" (professor.gmail.com) e clique em next;
    4. [S] O aplicativo deve deixar a borda do campo login vermelho e a mensagem "Por favor, digite um login válido" deve ser exibida;
    5. [E] Clicar em OK na mensagem;
    6. [S] O cursor de digitação deve estar no campo de login e o usuário deve poder corrigir o login anterior;
  5. Smartphone com o aplicativo instalado;
  6. Nenhum.
  7. Não (afinal, não preciso ter feito nenhuma ação antes de testar o login, já que é a primeira coisa que deve ser feita);

Reportando Defeito

O defeito depende de padrões definidos em cada empresa ou time de desenvolvimento.

Ferramentas utilizadas para gerenciar o projeto e a atividade de testes: Jira, Bugzilla ou testLink.

  • Identificador do Defeito: facilita a comunicação e localização;
  • Descrição: uma descrição sucinta do defeito encontrado;
  • Versão do Produto: para saber em qual versão do sistema o defeito foi encontrado;
  • Passos Detalhados: incluindo dados utilizados, Screenshots e vídeos para encontrar o defeito;
  • Data de Reporte do Defeito: para facilitar o gerenciamento;
  • Reportador por: para saber qual o testador identificou o defeito;
  • Status: novo/aberto, designado, em verificação, resolvido, fechado, falhado, não será corrigido, etc;
  • Corrigido por: nome do desenvolvedor que corrigiu o defeito;
  • Data de Encerramento: a data que o defeito foi dado como inexistente;
  • Severidade: bloqueia a versão. Crítico, pequeno;
  • Prioridade: alta, média ou baixa;

Bug

Anomalias/Problemas no software podem estar relacionados à perspectiva do desenvolvedor ou do usuário.

Independentemente de onde ocorra, é comumente chamado de BUG.

Um engano na introdução de dados leva a um defeito. Isso produz um erro na execução, faz o resultado do teste ser um erro e a consequência é o tester achar que ocorreu uma falha.

Exemplo 03: Soma

                def soma(a, b):
                    return a * b 
                
                print( soma(1, 2) )
  1. Engano: o programador trocou o símbolo de "+" por "*";
  2. Erro: o programa vai executar a * b em vez de a + b;
  3. Defeito: o programa multiplica as entradas em vez de somá-las;
  4. Falha: a saída obtida 2 é diferente da saída esperada 3;
  5. Pensamento: Tem um defeito, um bug;
